# SleepySheep 통합 점검 문서
작성일: 2026-07-09

## 1. 문서 목적
이 문서는 SleepySheep 프로젝트의 현재 상태를 한 번에 점검하고, 저장용으로 남기기 위한 통합 메모다.  
제출용 보고서가 아니라, 개발자 본인이 직접 확인하고 수정 우선순위를 잡기 위한 내부 점검 문서다.

---

## 2. 전체 총평

SleepySheep는 전체적으로 **완성도 높은 정적 웹앱 프로젝트**에 가깝다.

좋은 점:
- 기능 수가 많고 앱 콘셉트가 통일되어 있음
- HTML / CSS / JS 구조 분리가 비교적 잘 되어 있음
- localStorage, service worker, canvas, ES Modules 등을 실제로 활용함
- 모닝콜, 수면 기록, 출석, 갤러리, 공방, 미니게임 등 기능 연결이 잘 되어 있음

현재 핵심 상태:
- 기능이 무너진 상태는 아님
- 구조도 크게 나쁘지 않음
- 지금 필요한 것은 대규모 재구현이 아니라 **마감 정리, 버전 통일, 문서 정리, 제출 품질 보완** 쪽임

한 줄 결론:
> 못 만든 프로젝트가 아니라, 거의 다 만들었고 마지막 정리 단계가 남은 프로젝트로 보는 게 맞다.

## 5. 강점 정리

### 5-1. 기능 통합도가 높음
단순 수면 기록 앱이 아니라 다음이 결합되어 있음.

- 수면 관리
- 모닝콜
- 출석 보상
- 양 성장
- 양털 수집
- 공방 꾸미기
- 갤러리
- ASMR
- 고민 상담
- 미니게임

즉, “수면 메이트 + 감성 힐링 + 성장형 상호작용”이라는 방향성이 비교적 분명하다.

### 5-2. 기술 사용이 다양함
- localStorage 저장
- service worker 사용
- Canvas 활용
- ES Modules 구조
- 정적 호스팅 친화적 설계

### 5-3. 모닝콜 테스트 보완 반영
settings.html 기준으로 1분 뒤 테스트 기능이 실제로 들어가 있음.

확인 포인트:
- 1분 뒤 테스트 버튼 존재
- wakeAlarm 저장 로직 존재
- saveMorningCallSchedule 호출
- setTimeout 60초 처리
- 테스트 후 index.html?morningcall=1 이동

즉, 예전의 “설정 페이지에 그대로 있으면 테스트 알림이 제대로 안 뜨는 문제”는 보완 시도가 실제로 반영된 상태다.

### 5-4. 날짜 처리 개선
storage.js 안에서 로컬 날짜 기준 키 생성 함수를 사용하는 점은 긍정적이다.

이건 UTC 기준 문자열 슬라이스보다 안전한 방식이라, 출석/기록 관련 날짜 처리 안정성에 도움이 된다.

## 6. 핵심 보완 포인트

### 6-1. 버전 표기 불일치
가장 먼저 정리해야 할 문제다.

현재 확인된 버전 표기:
- js/constants.js → 1.96.0
- pages/settings.html 기본 문자열 → 1.72.4
- deployment-test-report.md → 1.92.x 계열

즉, 코드/HTML/문서 버전이 서로 다르다.

#### 왜 문제인지
- 문서와 코드가 안 맞아 보임
- 제출 시 혼란을 줌
- JS 로딩 실패 시 구버전 문자열이 그대로 보일 수 있음

#### 권장 조치
- 최종 버전 번호 하나를 결정
- constants.js / settings.html / 보고서 전부 동일하게 수정

예시:
```html
<strong id="app-version"></strong>
```
또는
```html
<strong id="app-version">SleepySheep v1.96.0</strong>
```

---

### 6-2. 문서와 코드 기준 불일치
deployment-test-report.md는 현재 업로드된 실제 코드보다 예전 상태를 기준으로 작성된 것으로 보인다.

#### 의미
- 보고서는 과거 스냅샷
- 코드는 더 최신 상태

#### 조치
보고서는 최소한 아래를 다시 맞춰야 함.
- 현재 최종 버전
- 실제 보완된 항목
- 현재 배포 기준 링크/경로
- 남은 한계점

---

### 6-3. Gemini API 키 클라이언트 직접 사용
worry.js 기준으로 Gemini API를 브라우저에서 직접 호출하는 구조로 보인다.

#### 문제점
- API 키 노출 위험
- localStorage 또는 브라우저 네트워크 탭에 남을 수 있음
- 정적 데모에서는 가능하지만 실서비스 구조로는 취약함

#### 조치 방향
지금 당장 구조를 바꾸기 어렵다면 최소한 내부 메모/보고서에 다음 취지로 기록:
> 현재 AI 상담 기능은 정적 웹앱 구조상 클라이언트 호출 방식으로 구현되어 있으며, 상용 서비스 단계에서는 서버 프록시 방식으로 보완이 필요하다.

---

### 6-4. 개발자 옵션 노출
settings 쪽에 개발자용 기능이 존재하는 것으로 보임.

예상 기능:
- XP 추가
- 양털 추가
- 해금 처리
- 상태 강제 변경

#### 문제
- 시연에는 편하지만 제출본에서 보이면 치트 기능처럼 보일 수 있음
- 평가자 입장에서는 완성도가 떨어져 보일 수 있음

#### 조치 방향
선택지:
1. 제출본에선 숨김
2. 특정 조건에서만 노출
3. 시연용 기능임을 따로 문서화

---

### 6-5. home.html 품질 보완 여지
home.html은 리다이렉트 전용 페이지로 보이지만 sheep.html 대비 메타 정보가 단순하다.

#### 보완 추천
- `lang="ko"`
- `charset`
- `viewport`
- `title`
- `location.replace(...)`

기능 오류급 문제는 아니지만, 제출 품질상 다듬으면 좋다.

---

### 6-6. 배포 경로 이슈 문서화 필요
기존에 언급된 `PBL1/PBL1` 경로 문제는 소스 일부만으로는 완전 해결 여부를 확정하기 어려웠다.

#### 의미
- 실제 배포 구조가 문서와 다를 수 있음
- 사용자 접근 링크가 복잡할 수 있음

#### 조치
최종 저장용 메모에 아래 체크 필요:
- 실제 배포 URL 1개만 사용 가능한지
- 중복 폴더 경로가 없는지
- index 기준 진입이 일관적인지

---

### 6-7. .agents 폴더 실내용 확인 필요
.agents.zip은 사실상 비어 있는 상태로 보였음.

#### 가능성
- 단순 빈 폴더를 같이 압축한 것
- 누락된 파일이 있을 수 있음

#### 조치
필요 없는 폴더면 정리하고, 필요한 파일이 원래 있었다면 재확인.

## 9. 지금 상태에 대한 최종 판단

현재 상태는 “기능이 부족한 프로젝트”라기보다,
“기능은 충분히 들어갔고 마지막 정리 단계가 남은 프로젝트”에 더 가깝다.

특히 좋은 점:
- 기능 수가 많다
- 앱 콘셉트가 살아 있다
- 기술 사용이 명확하다
- 정적 웹앱으로서 구현 범위가 넓다

주의할 점:
- 문서/버전/보안/제출 품질 디테일이 현재 제일 중요하다

즉, 앞으로의 작업 방향은:
- 기능 추가보다
- 정리 / 통일 / 문서화 / 마감 품질 보완

쪽이 맞다.
